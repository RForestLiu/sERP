## 三、模块架构

### 模块关系图

```
main.py (入口)
   │
   ├── app.py (Flask 后端) ← 所有 HTTP API 和路由
   │      │
   │      ├── 图片批量处理 (task_* 数据目录)
   │      ├── 产品采集 (/api/collect/*)
   │      ├── 产品管理 (/api/products/*)
   │      ├── Ozon 品类 (/api/ozon/*)
   │      │     ├── 品类树 (category-tree + 缓存)
   │      │     ├── 品类翻译 (translate-categories)
   │      │     ├── 品类匹配 (match-category → DeepSeek)
   │      │     └── 品类属性 (category-attributes)
   │      ├── Ozon 上架草稿 (/api/listings/*)
   │      ├── 店小秘自动填充 (/api/auto-fill/*)
   │      └── 店铺管理 (/api/stores/*)
   │
   └── collector.py (采集引擎)
          ├── 抓取层: Playwright + requests 双引擎
          ├── 决策层: DeepSeek 图片分类
          └── 执行层: aiohttp 异步下载 + Pillow 转 JPG
```

### 关键数据流

```
采集流程:
  URL → 抓取层 → product_data.json + image_urls
       → DeepSeek 分类 → images_mapping.json
       → 异步下载 → images/*.jpg

品类树翻译流程:
  Ozon API → category_tree.json (缓存)
           → 展平 7998 节点
           → 逐批发 DeepSeek → translations.json (缓存)
           → 前端合并渲染

Ozon 上架流程:
  产品数据 + 品类树 → DeepSeek 匹配品类
                    → 获取品类属性
                    → DeepSeek 填充属性值
                    → 保存上架草稿
```

## 四、日志系统

### 4.1 架构

| 文件 | Logger 名称 | 说明 |
|------|-------------|------|
| `app.py` | `logging.config.dictConfig` | ★ 唯一日志配置入口，在模块顶层执行 |
| `app.py` | `__name__` → `app` | 所有 API 路由日志 |
| `collector.py` | `__name__` → `collector` | 采集引擎日志 |
| `main.py` | — | 仅加载 .env + 启动 Flask，不配置日志 |

### 4.2 日志格式

```
HH:MM:SS | INFO  | app | 消息内容
HH:MM:SS | WARN  | app | 警告内容
HH:MM:SS | ERROR | app | 错误内容
HH:MM:SS | DEBUG | app | 调试内容（品类匹配等）
```

### 4.3 关键设计决策

- **为什么 `dictConfig` 放在 `app.py` 顶部而不是 `main.py`？**
  Flask `debug=True` 的 reloader 在 Windows 上通过 `subprocess.Popen` 创建子进程，子进程不会执行 `main.py` 顶层代码。放在 `app.py` 顶部保证无论哪个进程导入都能生效。
- **为什么需要 `sys.stdout.reconfigure(encoding='utf-8')`？**
  Windows 终端默认 GBK 编码无法输出 ❌ ✅ 等 emoji，会导致 `UnicodeEncodeError`，日志消息被静默丢弃。
- **`disable_existing_loggers: False`** — 防止 Flask/Werkzeug 已有的 logger 被禁用

### 4.4 改造记录（2026-04-27）

- `app.py`: 删除约 60 处 `print()`，改为 `logger.info/warning/error/debug`；删除 `app.run()`
- `collector.py`: 删除约 11 处 `print()`，改为 `logger`；修复文件结构损坏问题
- 新建 `main.py`: 统一初始化日志配置和 `.env` 加载，作为唯一入口

## 五、当前已知问题

### 🟡 问题 1：品类翻译偶有解析失败

**文件**: `app.py` → `_translate_categories_batch()`（ThreadPoolExecutor 并发翻译）

**症状**:
- 偶发某些批次 `✅ 成功解析 0 个翻译结果`
- DeepSeek 返回格式偶有波动，markdown 代码块包裹导致 JSON 解析失败

**当前状态**: 2026-04-27 已将 JSON 解析改为先清理 markdown 代码块再 `json.loads()`，大部分批次正常。仍有少数批次因为 DeepSeek 输出截断导致解析失败。

**后续改进方向**:
- 减小每批数量（当前 ~300-500，可降到 200）
- 增加 max_tokens（当前 32768）
- 失败批次自动重试

### 🟡 问题 2：品类匹配 DeepSeek max_tokens 偏小

**文件**: `app.py` → `ozon_match_category`，`max_tokens: 512`

当前品类匹配仅返回品类 ID，512 tokens 足够。如需返回更多属性建议，可调大到 1024。

### 🟢 已修复：品类匹配截断问题

删除了 `MAX_TREE_NODES = 2000` 截断逻辑，现在全量 7998 个品类发送给 DeepSeek，钱包等品类已可正确匹配。

### 🟢 已修复：字典值下拉列表不显示

`/v1/description-category/attribute/values` 调用始终传递 `type_id`，材质等字典属性可正确获取下拉选项。

### 🟢 已修复：属性描述为俄语

新增 `_translate_attr_descriptions` + `{store_id}_attr_desc_translations.json` 缓存，自动翻译属性描述为中文，hover 显示双语。

## 七、品类树缓存工作原理

```
Ozon API → 品类树 (7998 节点)
         → category_tree.json (原始缓存)
         → 展平所有节点
         → 检查 translations.json (id → 中文)
         → 未命中 → 分批发送 DeepSeek 翻译
         → 写入 translations.json
         → 前端合并渲染

关键文件:
  data/ozon_cache/{store_id}_category_tree.json
  data/ozon_cache/{store_id}_translations.json
```
